一、执行摘要

本报告对比 Open Code Review（OCR） 与 everything-claude-code（ECC） 两套体系中针对 Rust 代码的评审规则。两者并非同一类产品：OCR 将规则作为 prompt 注入统一审查流水线；ECC 则通过专用 subagent，在运行 cargo 工具链后再进行语义审查。

核心结论：OCR 在 Rust 语义深度（unsafe、async cancellation、FFI、API 设计）上更强；ECC 在工程门禁（check/clippy/fmt/test/audit）、严重级别划分与团队规范上更完整。最佳实践是将 ECC 的工具链门禁与 OCR 的语义规则结合使用。

六、优劣势总结

6.1 OCR 优势

  • 按文件类型自动匹配，无需额外命令
  • unsafe/async/FFI 等高级语义覆盖更深
  • 多语言统一审查流水线，输出可落地到 PR 行级评论
  • Cargo.toml 专项规则完整
  • 不依赖本地 Rust 构建环境即可做 diff 审查

6.2 OCR 劣势

  • 无 cargo check/clippy/test 硬门禁
  • 无严重级别与审批标准
  • 无 cargo audit 供应链漏洞检测
  • 缺少 thiserror/anyhow、文档、函数长度等工程规范
  • 纯 prompt 驱动，执行稳定性依赖模型

6.3 ECC 优势

  • 先跑工具链再审查，减少无效语义分析
  • Clippy/fmt/test/audit 形成确定性第一层过滤
  • 严重级别清晰，适合 Block/Warning/Approve 流程
  • Mutex poison、死锁、channel 等工程经验丰富
  • rust-patterns skill 示例丰富，适合 onboarding

6.4 ECC 劣势

  • 必须本地可构建，对 snippet/跨 crate 场景不友好
  • Rust 专用，难融入多语言统一审查
  • cancellation、原子序、FFI 等高级语义覆盖不如 OCR
  • 无行级评论定位体系
  • 依赖 Claude Code 生态（Bash/Read/Grep）

八、推荐组合方案

若希望兼顾精度与工程可执行性，建议采用分层审查：

  1. 第一层（ECC）：cargo check / clippy / fmt / test / audit 作为硬门禁
  2. 第二层（OCR）：rust.md 做 unsafe、async、FFI、API 设计等语义审查
  3. 第三层（OCR）：cargo_toml.md 审查 manifest、feature、MSRV、发布元数据
  4. 第四层（ECC）：严重级别、文档规范、函数长度等团队流程规则
